# GEO 网站构建模式

本文件记录从 geo.wiki 页面结构、机器可读索引和内容组织方式中提炼出的网页构建思路。用途：当 `wtt-seo-content-writer` 不只是写文章，而是要为一个站点、专题页、知识库或内容系统设计 GEO 友好结构时，按本文件执行。

## 1. GEO 友好网站的核心原则

一个 GEO 友好网站不是靠“多放 AI 关键词”实现的，而是靠四层结构同时成立：

1. **人能读懂**：页面有明确标题、速览、层级、段落、表格和链接。
2. **搜索引擎能收录**：静态或服务端可渲染 HTML、canonical、sitemap、robots、结构化数据完整。
3. **AI 能取用**：段落自包含、答案前置、内容块可独立引用、索引和导览对 LLM 友好。
4. **来源能复核**：每个非显然判断有来源、更新日期、作者/组织、许可和联系方式。

geo.wiki 的特征可以概括为：百科式内容架构 + 源头优先编辑规范 + 静态可抓取页面 + 机器可读索引。

## 2. 信息架构

### 2.1 顶层栏目

GEO 知识站适合采用“模块化百科”结构：

| 栏目 | 作用 | 页面类型 |
|---|---|---|
| Wiki / 百科 | 定义概念、术语、比较、标准、基础设施 | 概念页、术语页、对比页 |
| Playbooks / 操作手册 | 给出可重复执行的方法 | 审计、写作、追踪、实施流程 |
| Research / 研究 | 保存论文、基准、方法论证据 | 论文精读、benchmark、实验说明 |
| Platforms / 平台 | 拆解每个 AI 引擎机制 | ChatGPT Search、Perplexity、AIO、Gemini |
| Learn / 学习路径 | 把条目串成学习路线 | 入门、内容 GEO、技术 GEO、衡量、平台优化 |
| Pulse / 动态 | 追踪行业更新 | 官方动态、研究更新、平台公告 |

这个结构的 GEO 价值在于：每个页面都有明确的知识角色，AI 抓取后更容易判断页面是“定义”“方法”“证据”还是“平台说明”。

### 2.2 Hub-and-spoke 结构

站点应使用 hub 页面汇总子条目：

- `/wiki` 汇总所有概念条目。
- `/playbooks` 汇总所有操作手册。
- `/platforms` 汇总所有平台页。
- `/research` 汇总论文和基准。
- `/learn` 把核心条目组织成学习路径。

每个 hub 页面都应：

- 有一句清晰说明：这个栏目解决什么问题。
- 列出核心页面，每项带 1 句摘要。
- 使用稳定 URL。
- 链回相关概念、操作手册和平台页。

## 3. 单页内容结构

### 3.1 页面首屏

每个知识页首屏建议包含：

- 面包屑或栏目标签：告诉读者和机器页面类型。
- H1：一个明确概念或问题。
- 一句话总结：直接给定义或结论。
- 速览卡片：别称、领域、相关概念、核心杠杆、平台差异。
- 搜索或相关入口：方便人继续探索，也加强内部链接。

GEO 价值：AI 抓取页面前部时，就能拿到页面主题、定义、实体关系和上下文。

### 3.2 正文结构

正文按“答案块”组织：

1. 每个 H2/H3 第一段先给结论。
2. 重要概念给定义表或对比表。
3. 方法类内容用编号步骤。
4. 指标类内容用口径表。
5. 技术类内容写清“能做什么 / 不能做什么”。
6. 平台类内容写清类型、索引来源、爬虫/控制项、引用行为、优化重点。
7. 每页底部放相关条目、来源、更新时间。

### 3.3 FAQ

FAQ 应回答真实问题，不要为了 schema 硬造。

好的 FAQ：

- 问题本身就是用户可能问 AI 的句子。
- 答案 50-120 字，直接、完整、可独立引用。
- 不重复正文已有小标题。
- 与页面主题高度相关。

坏的 FAQ：

- “什么是本文？”“为什么它很重要？”这类空问题。
- 答案只是正文复制粘贴。
- schema 里有，页面可见正文没有。

## 4. 机器可读索引

### 4.1 sitemap

站点应提供 sitemap index 和具体 sitemap：

- `sitemap-index.xml`
- `sitemap-0.xml` 或按模块拆分的 sitemap

sitemap 中应包含：

- canonical URL。
- 中英文或多语言 alternate。
- 对真正更新的内容提供 `lastmod`。

GEO 价值：帮助宿主搜索索引发现和更新页面，间接影响 Google AI Overviews、Bing Copilot 等依赖搜索索引的答案面。

### 4.2 llms.txt

站点根目录应提供 `llms.txt`，作为面向 LLM 的精选导览。

推荐结构：

```markdown
# Site Name

> 一句话说明站点服务谁、解决什么问题。

## Modules

- [Wiki](https://example.com/wiki): 概念与定义
- [Playbooks](https://example.com/playbooks): 可执行方法
- [Research](https://example.com/research): 研究和证据
- [Platforms](https://example.com/platforms): 平台说明

## Key entries

- [核心概念](https://example.com/core): 一句话说明

## Machine-readable indexes

- [Full content index](https://example.com/llms-full.txt)
- [Sitemap](https://example.com/sitemap-index.xml)
- [robots.txt](https://example.com/robots.txt)

## License & contact
```

写法原则：

- 精选，不求全。
- 每个链接带一句解释。
- 指向 `llms-full.txt` 或模块索引，让愿意读取的工具继续深入。
- 写清许可、维护者、联系方式。

### 4.3 llms-full.txt

`llms-full.txt` 适合保存全站内容索引，不是全文堆砌。

推荐每条包含：

- URL。
- 标题。
- 所属模块。
- 1 句 shortDescription。
- 多语言版本链接。

GEO 价值：给 LLM 一个低噪声的全站地图，避免它从导航、页脚和重复 UI 中猜页面结构。

### 4.4 RSS

动态型站点应提供 RSS：

- 最新条目。
- 标题、链接、摘要、发布日期。
- 可按语言提供独立 RSS。

GEO 价值：让更新可被工具、聚合器和自动化流程订阅，形成新鲜度信号和再分发入口。

## 5. 结构化数据与 Head 元信息

### 5.1 基础 head

每页应具备：

- `<title>`：页面标题 + 站点名。
- `<meta name="description">`：80-160 字摘要。
- `<link rel="canonical">`。
- `hreflang` alternate。
- Open Graph / Twitter card。
- favicon、sitemap link、RSS link。

### 5.2 JSON-LD

按页面类型添加 JSON-LD：

- 首页：`WebSite`、`Organization`、可选 `SearchAction`。
- 概念页/文章页：`Article` 或 `TechArticle`，包含 headline、description、dateModified、author、publisher。
- FAQ：只有页面可见 FAQ 存在时添加 `FAQPage`。
- 面包屑：`BreadcrumbList`。

原则：

- JSON-LD 必须与页面可见内容一致。
- 不用 schema 伪造权威。
- 结构化数据是解析基础设施，不是直接引用杠杆。

## 6. 前端渲染与可访问性

### 6.1 渲染方式

GEO 站点优先使用 SSG 或 SSR。

要求：

- 主体正文在初始 HTML 中存在。
- 不依赖登录态或客户端接口才能看到核心内容。
- 页面在禁用 JS 时仍能读到标题、正文、链接和来源。
- 导航、主题切换、搜索增强可以用 JS，但不能承载唯一正文。

### 6.2 语义 HTML

页面应使用：

- 一个 `<main>`。
- 一个 H1。
- 顺序 H2/H3。
- 列表用 `<ol>` / `<ul>`。
- 表格用真实 `<table>`，表头完整。
- 引用和来源链接可被普通爬虫读取。

### 6.3 可访问性

可访问性也是机器可读性的基础：

- 跳到正文链接。
- 按钮有 `aria-label`。
- 搜索框有 label。
- 图片有 alt。
- 语言切换有 hreflang。

## 7. 内容编辑规范

### 7.1 Source-first

非显然判断需要来源。优先级：

1. 官方文档、论文、标准、厂商公告。
2. 可复核测量或公开数据。
3. 二手行业解读。

页面应能让读者追到原始出处，而不是只看到“业内认为”。

### 7.2 术语诚实

自定义概念要说明它是不是行业标准术语。

写法：

- 如果是行业通用术语，说明常见定义。
- 如果是本站为了组织知识提出的工作定义，明确写“工作定义”。
- 如果是近义词，说明边界和误用风险。

### 7.3 双语不是直译

多语言 GEO 站点应做“语义等深”，不是逐句翻译。

要求：

- 每个语言版本都用本地读者自然表达。
- 每个语言版本都有独立 canonical / hreflang。
- 关键术语保留原文和译名。
- 注意不同语言来源池和实体绑定可能不同。

### 7.4 No SEO theater

避免为了“看起来 SEO”而堆内容：

- 不为字数凑段落。
- 不为 schema 编 FAQ。
- 不为关键词重复标题。
- 不把来源堆在文末却不支撑正文判断。

## 8. GEO 站点页面模板

### 8.1 百科条目模板

```markdown
---
title: "[概念名]"
description: "[一句话定义]"
lastUpdated: YYYY-MM-DD
sources:
  - title: "[来源标题]"
    url: "https://..."
    tier: primary
---

# [概念名]

[一句话定义，直接回答它是什么。]

## 速览

| 项目 | 内容 |
|---|---|
| 别称 | ... |
| 所属领域 | ... |
| 解决什么 | ... |
| 不解决什么 | ... |

## 它在 Answer Loop 的哪一步起作用

[直接说明。]

## 与相邻概念的区别

| 概念 | 相同点 | 不同点 |
|---|---|---|

## 怎么做

1. ...
2. ...
3. ...

## 常见问题
```

### 8.2 操作手册模板

```markdown
# [任务名]

[一句话说明这份手册产出什么。]

## 输入

- 需要哪些页面、数据、日志或权限。

## 输出

- 一份什么格式的结果。

## 步骤

1. ...
2. ...
3. ...

## 判断标准

| 检查项 | 通过 | 失败 | 修复方向 |
|---|---|---|---|

## 报告模板
```

### 8.3 平台页模板

```markdown
# [平台名]

[一句话说明平台类型和 GEO 关键事实。]

## 速览

| 项目 | 内容 |
|---|---|
| 类型 | SERP 内嵌 / 检索增强对话 / 原生答案引擎 |
| 检索来源 | ... |
| 关键爬虫/控制项 | ... |
| 引用行为 | ... |
| GEO 重点 | ... |

## 平台如何跑 Answer Loop

## 该平台该怎么优化

## 不能误解的地方
```

## 9. 设计风格与交互

GEO 知识站应优先服务阅读、检索和引用，不应做营销站式大 hero。

推荐：

- 清晰导航，低噪声布局。
- 内容优先，首屏能看到定义和速览。
- 用卡片承载可扫描的条目，但不要把正文藏在卡片堆里。
- 搜索入口明显。
- 深色/浅色主题可以有，但不能影响文本可读性。
- 页面底部保留来源、许可、RSS、sitemap、llms.txt。

避免：

- 首屏只有抽象口号，没有定义。
- 大量动画、渐变和装饰导致正文延后。
- 关键内容在展开组件里默认不可见。
- 只给人看，不给机器读的 canvas / 图片文字。

## 10. 构建检查表

上线前检查：

- [ ] 站点是否有清晰模块：Wiki、Playbooks、Research、Platforms、Learn。
- [ ] 每个页面是否有 H1、description、canonical。
- [ ] 主体内容是否在初始 HTML 可见。
- [ ] 是否有 sitemap-index.xml。
- [ ] 是否有 robots.txt。
- [ ] 是否有 llms.txt 和可选 llms-full.txt。
- [ ] 是否有 RSS 或更新索引。
- [ ] 是否有 JSON-LD，且与可见内容一致。
- [ ] 是否有 hreflang。
- [ ] 每个知识页是否有速览、对比、步骤或 FAQ。
- [ ] 每个非显然判断是否有来源。
- [ ] 页面是否避免纯营销话术和关键词堆砌。
- [ ] 是否有许可、联系方式、维护者说明。
