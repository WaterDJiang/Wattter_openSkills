# AI 洞察分析 Prompt 模板

本文件定义了 App Store 评论洞察分析的所有 Prompt 模板。AI 助手在执行分析时，应严格遵循这些模板的结构和要求。

---

## 1. 核心洞察挖掘（mineReviewCorpus）

这是最核心的分析模板，用于从评论语料中挖掘结构化产品洞察。

### 输入格式

向 AI 提供以下信息：

```
App：{appName}
国家/地区：{countryName}
统计：{summaryJSON}
评论样本：{reviewsJSON}
```

其中：
- `summaryJSON`：来自 `summarize.py` 的 `summary` 字段（评分分布、版本分布等）
- `reviewsJSON`：来自 `summarize.py` 的 `sortedReviewsForAnalysis` 字段（最多 100 条，差评优先）

### Prompt 模板

```
你是一名资深 App 产品研究员。请基于 App Store 用户评价做信息挖掘。

App：{appName}
国家/地区：{countryName}
统计：{summaryJSON}
评论样本：{reviewsJSON}

请对以上评论进行深度分析，输出以下 7 个维度的结构化洞察：

## 1. 概览摘要（executiveSummary）
3-5 句中文总结，包含：
- 整体口碑基调
- 主要矛盾或冲突点
- 值得关注的产品机会

## 2. 痛点分析（painPoints）
用户最抱怨的问题，每条包含：
- **标题**：≤14 字，描述痛点本质
- **摘要**：1 句话说明影响
- **证据**：从原评论中摘取 ≤60 字的原文证据
- **优先级**：high / medium / low

要求：不要使用"问题""优化""改进"等口水词做标题，标题必须具备可理解的产品含义。

## 3. 产品机会（opportunities）
从差评中提炼的产品改进机会，格式同上。
要求：聚焦"用户在期待什么但目前没做好"，而非"修复什么"。

## 4. 正面信号（positiveSignals）
用户认可的价值点，格式同上。
要求：这些是产品的护城河，可强化和放大。

## 5. 用户分层（userSegments）
不同用户群体的差异化诉求，格式同上。
要求：按用户角色或使用场景分层，说明每群的独特需求和痛点。

## 6. 版本风险（versionRisks）
版本更新引发的性能/体验/定价风险，格式同上。
要求：关联具体版本号，说明版本变化带来的风险。

## 7. 行动建议（actionPlan）
可直接执行的产品动作，格式同上。
要求：
- 标题是一句可执行动作（而非问题描述）
- 证据说明"为什么要做"
- 按优先级排序

## 8. 深挖问题（queryAngles）
2-3 个后续值得深挖的问题，每条 ≤18 字。

---

### 严格要求

1. **所有输出用中文**
2. **每类洞察 2-3 项**，优先保留高频和高风险
3. **不要虚构评论里不存在的事实**，证据必须来自评论标题或内容
4. **只有评论确实没有相关证据时**，对应板块才可以为空
5. **不要因为样本少而返回空洞察**——即使只有少量评论，也要尽力提取有效信息
6. **证据必须来自提供的评论**，不可凭空编造用户言论
```

## 2. 主题提取（summarizeThemes）

用于从好评或差评中提取核心主题。

### Prompt 模板

```
你是一名资深产品分析师。请阅读一批用户评论（标题+内容），提取{polarity}核心主题。

任务：提取{polarityLabel}主题，给出中文短标题（≤16 字）与 2-3 句中文解读，并提供来自原文的一条证据句。

严格要求：
- 所有输出中文
- 不要使用"建议/问题/优化/改进"等口水词做标题
- 标题必须具备可理解的产品含义
- 证据句必须取自提供的原文内容或标题
- 最多返回 {limit} 个主题

输入评论：
{reviewsJSON}

请按以下 JSON 格式输出：
{
  "themes": [
    {
      "title": "中文短标题",
      "summary": "2-3 句中文解读",
      "examples": [{"id": "评论ID", "snippet": "证据句"}]
    }
  ]
}
```

其中：
- `polarity` = "positive" → `polarityLabel` = "好评核心亮点"
- `polarity` = "negative" → `polarityLabel` = "差评核心问题"

## 3. 问题分类与建议提取（summarizeIssuesSuggestions）

用于从评论中提取问题分类分析和改进建议。

### Prompt 模板

```
你是一名资深产品分析师。请阅读一批用户评论（标题+内容），只返回结构化 JSON。

任务：
1. 提取"问题分类分析"：用中文短标题（≤16 字）+ 1-2 句中文解读，按类别归属，并提供来自原文的一条证据句（含评论 ID）。
   类别选项：性能 | 功能 | 体验 | 内容 | 账户 | 价格 | 其他
2. 提取"改进建议"：仅当评论明确提出"希望/需要/增加/修复"等，给出中文短标题（≤16 字）+ 1-2 句中文解读，并提供证据句。

约束：
- 所有输出中文
- 不要使用"建议/问题/优化/改进"等口水词做标题
- 标题必须可行动且具体
- 每项提供 1 条证据句，来自输入原文
- issues 与 suggestions 各最多返回 {limit} 项

输入评论：
{reviewsJSON}

请按以下 JSON 格式输出：
{
  "issues": [
    {
      "title": "",
      "summary": "",
      "category": "性能|功能|体验|内容|账户|价格|其他",
      "examples": [{"id": "", "snippet": ""}]
    }
  ],
  "suggestions": [
    {
      "title": "",
      "summary": "",
      "examples": [{"id": "", "snippet": ""}]
    }
  ]
}
```

## 4. 单条评论分析（analyzeReview）

用于对单条评论进行深度分析（可选，适用于重要评论的逐一解读）。

### Prompt 模板

```
你是一名严谨的中文产品分析师。请分析以下用户评论，输出结构化 JSON。

评论标题：{title}
评论内容：{content}
评分：{rating}
版本：{version}
评论者：{authorName}
日期：{updated}

请输出 JSON：
{
  "sentiment": "positive|negative|neutral",
  "issues": ["具体问题描述1", "具体问题描述2"],
  "suggestions": ["具体改进建议1", "具体改进建议2"],
  "versionRefs": ["关联的版本号"]
}

要求：
- sentiment 基于评分和内容综合判断
- issues 提取用户明确抱怨的问题，每条 ≤16 字
- suggestions 仅提取用户明确提出或强烈暗示的改进建议
- versionRefs 提取评论中提及的版本号
```

## 5. 分析参数参考

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| 评论样本上限 | 100 条 | 差评优先排序 |
| 每条评论内容截断 | 700 字 | 防止 payload 过大 |
| 每类洞察项数 | 2-3 项 | 优先高频和高风险 |
| 证据句上限 | 60-220 字 | 保持精炼 |
| 问题分类体系 | 7 类 | 性能/功能/体验/内容/账户/价格/其他 |
| 主题提取数量 | 5 个 | 好评/差评各 5 个核心主题 |
