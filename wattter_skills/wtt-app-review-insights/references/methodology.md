# 方法论文档

本文档描述 App Store 评论洞察分析的方法论基础和设计原则。

---

## 核心理念

### 证据驱动（Evidence-First）

每条 AI 洞察都必须附带原始评论证据，避免空洞的通用结论。

- ✅ "响应速度慢" — 证据：用户原话 "每次提问都要等十几秒才出第一个字"
- ❌ "用户体验有待提升" — 没有具体证据，属于空洞结论

### 差评优先（Negative-First）

排序策略：差评（1-2 星）排在前面，然后按日期倒序。理由：

1. 差评包含更多可行动的产品洞察
2. 好评的信号通常比较一致，少量样本即可提取
3. 差评的多样性更高，需要更多样本来识别模式

## 数据获取策略

### 双源互补

| 维度 | Apple RSS | App Store HTML |
|------|-----------|----------------|
| **数据结构** | 结构化 JSON | 非结构化 HTML → JSON 提取 |
| **版本信息** | ✅ 有 | ❌ 无 |
| **分页支持** | ✅ 最多 20 页（~1000 条） | ❌ 仅首屏 |
| **国家覆盖** | 单一国家 | 可多国家补充 |
| **稳定性** | 较高 | 受 Apple 反爬影响 |
| **评论数** | 主力 | 补充 |

### 抓取流程

```
1. Apple RSS 分页抓取 → 最多 20 页
2. 如果 RSS 数量不足 → App Store HTML 多国补充
3. 合并去重（基于评论 ID）
4. 截取至 maxReviews
```

### App 解析流程

```
用户输入 → 判断类型
  ├─ App Store URL → 解析 ID + 国家
  ├─ 纯数字 → 作为 App ID 查找
  └─ 文字 → iTunes Search API 搜索
```

## 统计分析体系

### 评分分布

5 星制评分的标准统计：
- 平均评分
- 1-5 星各星级的数量和占比
- 差评占比（1-2 星）
- 好评占比（4-5 星）

### 版本趋势

按版本聚合评论，计算每个版本的：
- 评论数量
- 平均评分
- 差评/好评占比
- 情感分布

用途：识别版本质量退化、版本间评分波动。

### 问题热力图

6 大问题主题 × 版本 的交叉矩阵：

|  | 性能稳定 | 功能体验 | 价格限制 | 准确质量 | 账号权限 | 服务信任 |
|--|---------|---------|---------|---------|---------|---------|
| v1.0 | 15% | 10% | 5% | 8% | 3% | 2% |
| v2.0 | 25% | 12% | 20% | 6% | 4% | 5% |

用途：定位哪个版本的哪类问题最突出。

### 情感时间线

按日聚合评论情感（正面/中性/负面），展示最近 21 天趋势。

用途：识别差评峰值日，回查当天的更新/故障/策略变化。

### 诊断洞察

从统计数据中自动提炼的 4 条关键洞察：
1. **样本覆盖**：分析所用的评论样本量
2. **高风险版本**：差评占比最高的版本
3. **主导痛点**：出现频率最高的问题主题
4. **差评峰值**：差评最多的日期，或评分最高的正向版本

## AI 分析框架

### 6 维度洞察体系

| 维度 | 核心问题 | 输出 |
|------|---------|------|
| **痛点** | 用户最抱怨什么？ | 结构化痛点 + 优先级 + 证据 |
| **机会** | 用户在期待什么？ | 产品改进机会 + 优先级 + 证据 |
| **正面信号** | 用户认可什么？ | 产品护城河 + 优先级 + 证据 |
| **用户分层** | 不同用户群要什么？ | 用户群体 + 差异化诉求 + 证据 |
| **版本风险** | 哪个版本出了问题？ | 版本号 + 风险描述 + 证据 |
| **行动建议** | 应该先做什么？ | 可执行动作 + 理由 + 优先级 |

### 分析约束

1. **不虚构**：所有洞察必须有评论证据支撑
2. **不空泛**：标题 ≤14 字，摘要 1 句话，证据 ≤60 字
3. **不遗漏**：即使样本少，也要尽力提取有效洞察
4. **可行动**：行动建议必须是"做什么"而非"有什么问题"

### 问题分类体系

7 大问题分类（性能稳定、功能体验、价格限制、准确质量、账号权限、服务信任、其他），详见 `analysis_prompts.md` 第 3 节。关键词匹配逻辑在 `scripts/summarize.py` 的 `ISSUE_THEMES` 常量中定义。

## 适用场景

| 场景 | 如何使用 |
|------|---------|
| **竞品分析** | 对比多个同类 App 的评论洞察 |
| **版本复盘** | 关注版本趋势和版本风险维度 |
| **用户调研** | 关注用户分层和正面信号维度 |
| **产品规划** | 关注产品机会和行动建议维度 |
| **危机预警** | 关注差评峰值和高风险版本 |

---

## 局限性

1. **评论数量**：Apple RSS 最多获取约 1000 条评论，HTML 补充有限
2. **版本信息**：HTML 来源的评论没有版本信息（标记为 "Unknown"）
3. **语言覆盖**：主要优化中英文评论的分析
4. **时效性**：评论数据为抓取时刻的快照，不实时更新
5. **反爬风险**：Apple 可能限制高频请求，建议控制抓取频率
