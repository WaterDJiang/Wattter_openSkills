# 合并与增量规则

第十一步「合并模式」的完整规则。这是把 skill 从「一次性蒸馏」升级为「编译器」的关键差异。

## 触发条件

第零步读 index 时，如果发现以下任一情况，本次编译进入"合并模式"：

1. wiki/ 下已存在与本次拟产卡片同名的文件
2. 本次输入资料与 log.md 中某条记录的"源"重叠（同一份文件二次编译）
3. 用户显式说"补充/更新/扩展" wiki

## 合并的四种情况

### 情况 1：完全同名同义

**条件**：`kp_name` + `module` + `knowledge_domain` 全部一致，且语义相同（不是巧合同名）。

**处理**：

1. 不覆盖原文件
2. 在原文件追加新发现的素材、边界、反例、关联
3. frontmatter 增加 `last_compiled: {本次日期}`，保留原 `source_marking`
4. 如果新内容与旧内容**冲突**，用以下块并存：

```markdown
> ⚠️ 与上次产出冲突（{旧日期} vs {新日期}）：
> - 旧版：{原表述}
> - 新版：{新表述}
> - 处理：{并存 / 待用户裁决 / 已合并}
```

5. 在 log.md 的"冲突"字段点名

**不要做的事**：不要"智能合并"两个相反的论断——直接并存，让用户在下次 review 时决定。

### 情况 2：同名但异义（同名异义）

**条件**：`kp_name` 一致，但 `knowledge_domain` 或语义不同。

**例**：「四象限法」在时间管理（艾森豪威尔矩阵）和波士顿矩阵里都有，但完全是两个东西。

**处理**：

1. 新文件加领域后缀：`四象限法--时间管理.md` vs `四象限法--战略分析.md`
2. 在两个文件的"关联"段互写：`> 同名异义：见 [四象限法--{另一领域}](./四象限法--{另一领域}.md)`
3. index.md 的领域索引按各自领域分别列出
4. 如果原文件没有后缀（先存在的文件），**询问用户**是否一并加后缀，还是保留无后缀的为"主版本"

### 情况 3：跨课程同义（应抽出到 concepts/）

**条件**：同一概念在多个课程中独立出现，且语义一致（如"MECE 原则"在选题、咨询、汇报三个课程中都被讲到）。

**处理**：

1. 在 `wiki/concepts/{概念名}.md` 创建概念卡片，frontmatter：

```yaml
---
concept_name: MECE 原则
knowledge_domain: 通用方法论
knowledge_type: 原则
appears_in:
  - wiki/kps/选题方法论-knowledge-points/01-选题/MECE原则.md
  - wiki/kps/咨询基础-knowledge-points/02-分析/MECE原则.md
  - wiki/kps/高效汇报-knowledge-points/03-结构/MECE原则.md
source_marking: 已提取
---
```

2. 概念卡片 body：通用定义 + 各课程中的不同侧重（短描述 + 链回原 kp）
3. 各课程的 kp 文件**保留**，但在头部加：

```markdown
> 📚 此知识点的通用定义见 [wiki/concepts/MECE原则.md](../../../concepts/MECE原则.md)
> 本文聚焦「{本课程视角}」
```

4. log.md 的"跨课程归并"字段记录

**何时不抽**：如果概念在不同课程中有**实质性的视角差异**（不只是侧重不同），保持各自独立，不强行归并。

### 情况 4：source_marking 升级

**条件**：上次编译标记为 `[AI补全]`，本次原文找到了依据，应升级为 `[已提取]`。

**处理**：

1. 直接更新 frontmatter 的 `source_marking: 已提取`
2. 增加 `upgraded_at: {本次日期}`
3. 在 body 对应行更新标记
4. log.md 不必单独记录（不算冲突），但可在"新增"字段写"3 项标记升级"

## 增量 ingest 的边界

### 应该增量的情况

- 同一课程的补充资料（新录音稿 / 新参考文章）
- 同一案例的后续追踪
- 用户主动说"基于这份新材料更新"

### 不应该增量的情况（应另起一个产出）

- 输入资料的核心论点与已有产物冲突 → 让用户决定是更新还是新建
- 跨度过大（比如时隔 1 年的同名课程改版）→ 建议用版本后缀，如 `选题三维度--v2.md`，避免污染历史

## 去重检查清单

进入合并模式后，编译第一张卡片前，过一遍：

- [ ] 已读 wiki/indexes/index.md 的领域索引和模块表
- [ ] 已 ls 目标输出目录，列出现有文件
- [ ] 对每张拟产卡片，匹配现有文件名 + 同义词
- [ ] 同义词识别（举例）：
  - 「MECE 原则」≈「相互独立完全穷尽」≈「MECE」
  - 「金字塔结构」≈「金字塔原理」≈「Pyramid Principle」
  - 「四象限法」≠「四象限矩阵」（要看具体内容）
- [ ] 拿不准是否同义 → 问用户

## 合并时的产出格式

合并后的卡片结构：

```markdown
---
kp_name: 知识点名称
module: 所属模块
knowledge_domain: 领域
knowledge_type: 类型
tags: [...]
source_marking: 已提取
source_files:
  - raw/articles/初版.md          # 上次
  - raw/transcripts/补录.md       # 本次
last_compiled: 2026-06-15
upgraded_at: 2026-06-15
---

# {知识点名称}

> 一句话描述

## 概览
| 卖点 | 笑点 | 特点 | 焦点 |
| ... |

## 边界追问
| 前提假设 | 适用边界 | 反例/盲区 |
| ... |

> ⚠️ 与上次产出冲突（2026-05-26 vs 2026-06-15）：
> - 旧版：{表述}
> - 新版：{表述}
> - 处理：并存——前者适用 X 场景，后者适用 Y 场景

## 可用素材
| 类型 | 内容 | 来源 |
| 案例 [已提取] | 旧素材 | 初版.md |
| 案例 [已提取] | 新素材 | 补录.md |
...

## 关联
（双向关联，新增的关联也写进对应的 B 卡片）
```

## 不要做的事

- ❌ 不要"为了好看"重写已有卡片的旧内容——append 即可
- ❌ 不要在没有用户确认的情况下批量重命名（同名异义后缀化、跨课程归并）
- ❌ 不要静默丢弃冲突——必须显式记录
- ❌ 不要忘记 log——合并模式的 log 比新建模式更重要（追溯链）
