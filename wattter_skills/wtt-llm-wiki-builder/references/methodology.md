# 蜂窝矩阵（四点三线）方法论参考

## 概述

蜂窝矩阵是课程模块设计的立体框架，确保每个模块既有记忆点，又有逻辑力，还有情绪张力。

## 四点

每个知识模块必须回答四个问题：

| 点 | 含义 | 设计问题 | 示例 |
|:---|:---|:---|:---|
| **卖点** | 课程的价值主张 | 学员为什么要花时间听这课？ | "学完这个方法，你的方案通过率提升3倍" |
| **笑点** | 情绪释放点 | 这节课哪里能让学员笑/松一口气？ | 一个自嘲、一个反差案例、一个趣味视频 |
| **特点** | 差异化标签 | 这节课跟别的课有什么不同？ | 独家模型、首创工具、独家案例 |
| **焦点** | 核心记忆点 | 学员离开时最应该记住的一个东西是什么？ | 一个模型、一句金句、一个动作 |

### 四点萃取指南

- **卖点提取**：关注内容中反复强调的价值承诺、收益描述、解决的问题
- **笑点提取**：关注幽默元素、反差对比、自嘲段落、趣味案例
- **特点提取**：关注独家方法、首创概念、差异化表述、跟行业惯例不同的做法
- **焦点提取**：关注反复出现的核心概念、金句、唯一要记住的原则

### 四点补全原则

- **卖点**必须具体可量化，避免"提升效率"这种空话
- **笑点**必须自然，不能为搞笑而搞笑。优先考虑反差、自嘲、反常识
- **特点**必须有对比参照系——"跟谁比不一样？"
- **焦点**必须"一句话能转述"——学员离开时能脱口而出

## 三线

贯穿整个课程的三条暗线：

| 线 | 含义 | 设计要求 | 检验标准 |
|:---|:---|:---|:---|
| **情绪线** | 课程的情绪起伏节奏 | 基于峰终理论，每8分钟设计一个波峰 | 情绪体验图：有波峰有波谷，不是一条直线 |
| **逻辑线** | 知识的递进结构 | 从已知到未知、从简单到复杂、从问题到方案 | 删除任意一个模块，其余是否仍能支撑主题？ |
| **哲理线** | 可迁移的底层洞察 | 从具体案例上升到普适原则 | 学员能否把课堂方法迁移到其他场景？ |

### 三线萃取指南

- **情绪线提取**：分析内容的节奏变化（哪里紧张、哪里放松、哪里高潮）
- **逻辑线提取**：分析知识的递进关系（概念A→概念B→概念C的推导链）
- **哲理线提取**：分析可迁移的洞察（哪些结论不仅适用于当前场景）

### 三线补全原则

- **情绪线**：运用峰终理论，确保开场有钩子、中段有起伏、收尾打高终峰
- **逻辑线**：确保每一步推导都有"为什么是这个顺序"的理由
- **哲理线**：从具体到抽象，提炼出一句话级别的可迁移洞察

## 情绪线与四点的节奏映射

课程不同阶段，四点各有侧重：

| 阶段 | 情绪 | 技巧 | 四点侧重 |
|:---|:---|:---|:---|
| 开场 | 好奇/期待 | 悬念、痛点、数据 | 焦点+卖点 |
| 中段 | 投入/思考 | 互动、讨论 | 笑点+特点 |
| 高潮 | 兴奋/顿悟 | 最佳案例、核心洞察 | 卖点+特点 |
| 收尾 | 满足/行动 | 金句、立刻可执行的CTA | 焦点+哲理 |

## 面条理论支撑

- **面条（干货）**：专业知识、原理说明——服务于教学目标，可能位于情绪线波谷
- **佐料（素材/活动）**：案例、故事、视频、游戏——服务于情绪线，把波峰值打高
- **节奏法则**：每8分钟必须有一个素材切换点

## 知识点判定标准

一个合格的知识点必须同时满足：

1. **独立性**：可以单独拿出来讲，不依赖其他知识点的上下文也能听懂
2. **可操作性**：学员学完后能独立运用或转述，不是纯背景信息
3. **粒度适中**：讲授时间约8-20分钟，不过粗（一个模块就是一个知识点）也不过细（一句话就是一个知识点）

### 知识点 vs 模块

| | 模块 | 知识点 |
|:---|:---|:---|
| **粒度** | 解决一个大问题 | 解决一个子问题 |
| **数量** | 3-7个/课程 | 按实际内容识别，宁缺毋滥 |
| **时长** | 1-3小时 | 8-20分钟 |
| **结构** | 包含零到多个知识点 | 不可再分的最小教学单元。如果模块中没有可独立讲授的知识点（如纯练习/辅导环节），则不强行拆分 |

### 拆解示例

模块"AI选题破题"→拆解为：
- 知识点A：**选题三维度**（稀缺度/实用度/鲜活度）——判断课题值不值得做
- 知识点B：**选题四字诀**（小/实/专/新）——缩小课题范围的四个操作原则
- 知识点C：**四维度缩小法**——从四个维度把大课题缩小到可交付范围
- 知识点D：**SKA分析法**——从技能/知识/态度三维度拆解教学目标
- 知识点E：**布鲁姆教学目标分类与ABCD撰写法**——写出可衡量的教学目标

## 边界追问指南

对每个知识点进行批判性审视，划定有效范围：

### 追问维度

| 维度 | 追问问题 | 示例 |
|:---|:---|:---|
| **前提假设** | 这个知识点依赖哪些隐含前提？换行业/换规模/换文化还成立吗？ | "MECE原则"依赖的前提是问题域可穷举——如果是开放式创意问题，MECE不适用 |
| **适用边界** | 什么条件下这个知识点会失效？上限和下限在哪？ | "四字诀缩小课题"在课题本身过窄（已无缩小空间）时不适用 |
| **反例/盲区** | 原文没提但实践中容易踩的坑？有没有反面案例？ | "AI优化大纲"的盲区：AI可能把差异化内容优化成行业通用模板 |

### 质疑原则

- 质疑不是否定，而是划定有效范围——让知识更精确
- 如果原文没有提供边界信息，基于逻辑推断并标记 `[AI补全]`
- 好的边界追问让知识点**更可信**，而非削弱它

## 素材标记指南

从原文中识别并分类可复用的教学素材：

### 素材类型

| 类型 | 识别特征 | 标记要点 |
|:---|:---|:---|
| **案例** | 真实业务场景、成功/失败故事、客户案例 | 标注行业和场景，一句话概括案例核心冲突 |
| **数据** | 统计数字、调研结果、对比数据、趋势数据 | 标注数据来源和时间范围 |
| **练习** | 行动学习环节、互动讨论、课后作业、模拟演练 | 标注练习形式和预计时长 |
| **话术** | 金句、口诀、表述模板、可复用的开场/结尾词 | 标注适用环节（开场/中段/收尾） |
| **工具** | 模板、清单、检查表、模型图、流程图 | 标注工具用途和使用时机 |

### 标记原则

- 原文已有的素材直接提取并标注类型
- 原文缺失但该知识点明显需要某类素材时，用 `[AI补全]` 给出建议方向
- 一个知识点通常不需要填满所有五种类型——只列实际存在或真正需要的

## 知识点关联指南

建立知识点之间的多维关联网络：

### 关联类型

| 类型 | 含义 | 标注格式 |
|:---|:---|:---|
| **前置依赖** | 知识点B的正确理解依赖于先掌握知识点A | `→ [[模块X.知识点Y]]：前置依赖` |
| **互补关系** | 两个知识点互相补充，合在一起才能完整解决问题 | `↔ [[模块X.知识点Y]]：互补` |
| **跨域类比** | 该知识点在其他领域有结构类似的模型 | `≈ [领域]的[模型名]：类比说明` |
| **教学迁移** | 该知识点的教学方法可复用到其他主题 | `⇒ 可迁移至：[场景描述]` |

### 关联原则

- 只标注有实质关系的关键关联，不要为了关联而关联
- 跨域类比尤其有价值——帮助学员用已知领域的经验理解新知识
- 如果原文没有显式关联，但存在明显的结构相似性，可用 `[AI补全]` 标注

## 知识标签体系

为支持渐进式引用和跨课程关联，每个知识点必须标注 `knowledge_domain`（领域标签）和 `knowledge_type`（知识类型标签）。

### 领域标签（knowledge_domain）

标识知识点所属的行业/专业领域，用于粗筛和跨课程关联。

| 领域 | 说明 | 典型知识点举例 |
|:---|:---|:---|
| **AI** | 人工智能、机器学习、大模型相关 | Prompt工程、RAG架构、微调策略 |
| **营销** | 市场营销、品牌、增长 | 选题三维度、爆款公式、用户画像 |
| **管理** | 组织管理、项目管理、团队协作 | OKR制定、敏捷流程、授权模型 |
| **工具使用** | 软件工具、平台操作、自动化 | Notion搭建、n8n工作流、飞书集成 |
| **银行** | 银行业务、风控、合规 | 信贷审批流程、反洗钱规则、监管报送 |
| **金融** | 投资理财、财务分析、估值 | 财报解读、DCF估值、风控指标 |
| **教育** | 教学设计、课程开发、学习科学 | 布鲁姆分类、ABCD撰写法、渐进式披露 |
| **法律** | 合同、合规、知识产权 | 条款审查、风险点识别、协议模板 |
| **设计** | 视觉设计、UX、品牌视觉 | 配色理论、信息架构、组件库设计 |
| **通用** | 跨领域通用的思维模型和方法论 | MECE原则、第一性原理、结构化思维 |

> 以上为常见领域参考，不限于此。按课程实际内容标注，优先使用已有的领域标签保持一致性。一个知识点只标一个主领域。

### 知识类型标签（knowledge_type）

标识知识点的性质，用于同领域内排序和关联。

| 类型 | 说明 | 判定标准 | 示例 |
|:---|:---|:---|:---|
| **框架** | 多维度结构化的分析/决策工具 | 包含2个以上维度，有明确的坐标轴或矩阵 | 选题三维度、SWOT、四点三线 |
| **模型** | 抽象化的认知模型或理论模型 | 用简化的结构描述复杂现象 | 布鲁姆分类、SKA模型、冰山模型 |
| **原则** | 指导行为的准则或规律 | 一句话能判断对错/好坏 | 选题四字诀、剃刀法则、MECE原则 |
| **方法** | 有步骤的操作性方法 | 有明确的步骤序列 | ABCD撰写法、四维度缩小法、OKR制定法 |
| **流程** | 端到端的完整工作流程 | 从输入到输出的完整链路 | 课程开发八步曲、n8n工作流 |
| **工具** | 具体的工具/模板/检查表 | 拿来就能用的实体或数字工具 | Prompt模板、评估检查表、大纲模板 |
| **概念** | 需要理解但不能直接操作的定义性知识 | 定义、术语、分类体系 | 渐进式披露、峰终理论、token损耗 |

### 打标规则

1. **knowledge_domain**：只标一个主领域，选最直接相关的。如果知识点横跨多领域，选"主要服务于哪个领域的场景"
2. **knowledge_type**：只标一个主类型，按上表优先级选最具体的。框架>模型>原则>方法>流程>工具>概念
3. **tags**：自由补充标签，用于细粒度关联。可以标：行业-specific（如`银行`）、受众-specific（如`入门`）、难度（如`进阶`）、跨域关联词（如`结构化`）
4. 如果原文未明确说明领域/类型，基于知识点内容推断并标记 `[AI补全]`

### 渐进式引用

标签设计的核心目的是支持三层渐进加载，减少 token 损耗：

| 层级 | 加载内容 | 用途 | token 开销 |
|:---|:---|:---|:---|
| **L1 粗筛** | index.md 的领域索引表 | 按领域筛选相关知识点，决定是否深入 | 极小 |
| **L2 定位** | _module-summary.md + 知识点文件的 frontmatter + 概览表 | 理解知识点核心内容，判断是否需要详情 | 中等 |
| **L3 详情** | 完整知识点文件（边界追问、素材、关联） | 深入使用该知识点 | 全量 |

引用策略：先用 L1 找领域→再用 L2 看概览→仅加载需要的 L3。

## 使用场景

- **大纲设计阶段**：用四点检查每个模块是否"值得讲"
- **教案设计阶段**：用三线检查情绪节奏和逻辑递进
- **内容诊断阶段**（本skill核心场景）：对已有内容/课纲/想法进行四点三线提取和补全
